# 現実科学ラボ サイト — メンバー導入＆作業開始ガイド

このスキルは、参画メンバーが **Claude Code（または Codex）に話しかけるだけ**で、迷わず「最新の状態で作業を始める→記事を出す」までいけるように誘導する。人間向けの読み物は [docs/working-with-claude.md](../../../docs/working-with-claude.md)（ルール正本）と [docs/getting-started.md](../../../docs/getting-started.md)。

リポジトリ: `https://github.com/StudioSaitama-LLC/reality-science-site`（公開）。作業はクローンしたフォルダ（`package.json` がある所）の中で行う。

## まず伝える3つの合言葉（毎回これで突破できる）

1. **詰まったらモデルを上げる。** セットアップやトラブル対応は **Opus を Mid 以上**で。Sonnet/Haiku の Low は、文脈で解決せず迷宮に潜っていきやすい（「CPU が…」等の的外れに固執する）。
2. **コマンドは手打ちしない。** Claude が「ターミナルで以下を実行してください」と促してきたら、**「それ、あなたがやって」**と言って Claude 自身に実行させる。
3. **エラーは貼って「解消して」。** 赤いエラー文をそのまま貼って「これを解消して」と言えば進む。

---

## 使い分け

- **はじめての人** → 「A. 初回セットアップ」を一度だけ。
- **2回目以降、作業を始めるとき** → 必ず「B. 作業開始の同期チェック」から。

---

## A. 初回セットアップ（一度だけ）

### 1. クローンする
- ユーザーに置き場所（どのフォルダに置くか）を確認してから、リポジトリ URL を `git clone` する。**同名フォルダの二重作成に注意**（既に同名フォルダがあるとネストして「2つできた」状態になりやすい）。
- 落ちてきた `reality-science-site/`（`package.json` がある所）が作業ルート。Claude Code/Codex でそのフォルダを開く。
- `npm install` を実行（プレビュー・ビルドに必要）。

### 2. GitHub 認証（push できるようにする）
- push や PR 作成で **権限エラー（403）が出るのは想定どおり**。`gh`（GitHub CLI）が要る。
- 手順: `brew install gh` → `gh auth login`（"Login with a web browser" を選ぶ）→ ブラウザでデバイスコードを入力 → 端末と GitHub アカウントが紐づく。
- ⚠️ **パスワード入力（Mac のパスワード等）は画面共有を止めてから。** 共有中はターミナルにも表示され得る。
- 補足: StudioSaitama-LLC（将来はデジハリ org）の org メンバーは基本この認証なしで通る。**repo 単位の協力者だけ** `gh` 認証が要る。外部ライターには org 権限を渡さず、`.md` を受け取って管理者が上げる運用でもよい。

### 3. モデル／エフォートを上げておく
- 最初は **Opus Mid 以上**にしておく。記事追加のように `add-article` スキルで手順が固いタスクは軽いモデルでも通るが、セットアップ中は上げておくのが安全。

## B. 作業開始の同期チェック（毎回・最重要）

複数人が触るので、**古いローカルで作業すると競合・手戻りになる**。作業を始める前に、必ずローカルがリモートの最新と揃っているか確認する。ユーザーが「作業を始める」「最新にして」等と言ったら、まずこれを実行する。

```bash
git fetch origin
git status -sb        # 例: ## main...origin/main [behind 2]
```

`git status -sb` の1行目と作業ツリーの状態で判定する:

| 状態 | 見え方 | 対応 |
|---|---|---|
| 最新・クリーン | `## main...origin/main`（ahead/behind なし・変更なし） | そのまま作業開始してよい |
| 遅れている・クリーン | `[behind N]`、変更なし | `git pull --ff-only` で最新化してから開始 |
| 未コミットの変更がある | ` M ファイル` 等が出る | **勝手に捨てない。** 内容を要約して報告し、「前回の続きか／コミットするか／破棄するか」を本人に確認 |
| 未プッシュのコミットがある | `[ahead N]` | 報告（プッシュ済みか、PR にするか確認） |
| main 以外のブランチにいる | `## post/xxx...` 等 | どのブランチにいるか報告。新規作業なら main に戻って最新化してからブランチを切る |

- オープン PR の有無も確認: `gh pr list --state open`。あれば報告する。
- **最終的に「main が最新・クリーン」を確認してから、新しいブランチを切って作業に入る**（`main` への直 push は禁止。必ずブランチ → PR）。

> なぜやるか: スキルもルールも記事も全部この repo の git で管理されているので、最新を取らずに作業すると、他の人の変更を上書きしたり、古い前提で直してしまう。最初の `git fetch` 1回で防げる。

## C. 記事を追加して公開する

- 記事追加は **`add-article` スキル**に任せる（素材の Markdown/PDF/Google ドキュメント/テキスト ＋ 画像を渡すだけ）。詳細は [docs/adding-articles.md](../../../docs/adding-articles.md)。
  - **Vol 番号・開催日は必ず明示する**（指定しないと「最新+1」を推測する。例: 最新が 73 なら 74 を作ろうとする）。
  - **アイキャッチ画像が無いと HOME のグリッドに出ない**。
- 公開前に必ず**ローカルプレビューで確認**（`npm run dev -- --host` → 表示された permalink）。HOME 3領域の反映を確認したいときは dev を再起動するか `npm run build`（[adding-articles.md](../../../docs/adding-articles.md) 参照）。
- OK なら **ブランチ → PR**。**PR のマージ＝即公開**なので、むやみに押さない。練習は Vol.100/200 など適当な番号で行い、**関係ないテスト記事は削除してから**本番に進む。公開後でも Claude 経由で「さっきの無しで」と言えば revert できる。

## 困ったとき

1. **モデルを上げる**（Opus Mid 以上）。これだけで直ることが多い。
2. エラーを**そのまま貼って「解消して」**。
3. それでもダメなら **@Chiaki にスクリーンショットを貼って相談**。

## 守ること

- `main` に直接 push しない。push / PR は必ずブランチ経由。
- HOME（NEWS/NEXT LECTURE/グリッド）や `src/mirror/` を手で書き換えない（自動生成・忠実ミラー）。
- 公開リポなので**機密情報を置かない**（議事録など非公開のものは private repo `reality-science-docs` 側）。
