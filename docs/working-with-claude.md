# Claude Code で運用するメンバーのルール

現実科学ラボのサイト(reality-science-site)を **Claude Code / Codex 経由で**運用する人向けのルール正本。手順を自動で誘導してほしいときは Claude Code 同梱の **`/onboard` スキル**（初回セットアップ＋作業開始の同期チェック）と **`/add-article` スキル**（記事追加）を使う。Markdown を手で書く前提の全体像は [導入ガイド](getting-started.md) / [運用マニュアル](operations.md) を参照。

このルールは 2026-06-19 の定例で、実際に複数メンバーがセットアップ〜公開を実演し、**詰まった箇所**を元に作っている。

---

## 0. 3つの合言葉（まずこれ）

1. **詰まったらモデルを上げる** — セットアップ・トラブル対応は **Opus Mid 以上**。
2. **コマンドは手打ちしない** — 「ターミナルで実行して」と促されたら **「あなたがやって」** と Claude 自身に実行させる。
3. **エラーは貼って「解消して」** — 赤いエラー文をそのまま貼る。

---

## 1. モデル／エフォートの選び方（今日いちばん効いた発見）

| 用途 | 推奨 | 理由 |
|---|---|---|
| セットアップ・GitHub 認証・エラー解消・仕様変更 | **Opus Mid 以上** | Sonnet/Haiku の Low は文脈で解決せず迷宮に潜る（「CPU/メモリが…」等の的外れに固執）。実演でも Opus に上げた瞬間に解決した |
| 記事の追加（`add-article` で手順が固い） | Sonnet / Haiku でも可 | スキルがガードレールを強く縛っているため軽いモデルでも再現する |

合言葉は **「思ったように動かない＝まずモデルを上げる」**。スキルで固めたタスクほど軽いモデルで通り、曖昧な判断ほど賢いモデルが要る。

---

## 2. 作業開始時の同期チェック（毎回・必須）

複数人で触るので、**古いローカルのまま作業すると競合・手戻りになる**。作業を始める前に、ローカルがリモートの最新と揃っているか必ず確認する。Claude に「作業を始める前にリモートと同期チェックして」と言えば `/onboard` がこの手順を踏む。

```bash
git fetch origin
git status -sb
gh pr list --state open   # オープン PR の確認
```

判定:

- **最新・クリーン** → そのまま開始。
- **遅れている（`[behind N]`）・クリーン** → `git pull --ff-only` で最新化。
- **未コミットの変更がある** → 勝手に捨てず、内容を要約して「続きか／コミットか／破棄か」を確認。
- **未プッシュのコミット（`[ahead N]`）／作業ブランチにいる／オープン PR がある** → 報告してから判断。

最終的に **「main が最新・クリーン」を確認 → 新しいブランチを切る → 作業**。`main` への直 push は禁止。

## 3. GitHub 認証と権限

- push / PR で **権限エラー（403）が出るのは想定どおり**。GitHub CLI (`gh`) が要る: `brew install gh` → `gh auth login`（web browser / device flow）。
- ⚠️ **パスワード入力は画面共有を止めてから**（ターミナルにも出得る）。
- **org メンバー**（StudioSaitama-LLC、将来デジハリ org）は基本この認証なしで通る。**repo 単位の協力者だけ** `gh` 認証が必要。
- **外部ライター**には org 権限を渡さない。`.md` を受け取って管理者が上げる運用でよい。

---

## 4. 記事の追加〜公開（安全弁）

- 入口は **`/add-article`**：素材（Markdown/PDF/Google ドキュメント/テキスト）＋画像を渡すだけ。詳細は [adding-articles.md](adding-articles.md)。
- **Vol 番号・開催日は必ず明示**（指定しないと「最新+1」を推測する）。
- **アイキャッチ画像が無いと HOME グリッドに出ない**。
- 公開前に**ローカルプレビューで確認** → OK なら **ブランチ → PR**。
- **PR のマージ＝即公開**。むやみにマージしない。練習はテスト番号（Vol.100/200 等）で行い、**関係ないテスト記事は削除**してから本番へ。公開後でも Claude 経由で「さっきの無しで」と言えば revert できる。

---

## 5. HOME の自動更新（混乱しやすい点）

- 記事を1本足すと **NEWS / NEXT LECTURE / レクチャーグリッド**は自動更新される（手で直さない）。
- ただし **dev サーバーを起動したまま記事を足すと、HOME 3領域がホットリロードに乗らず古いまま**のことがある → **dev を再起動**するか **`npm run build`** で確認（[adding-articles.md](adding-articles.md) 参照）。

---

## 6. スキルの育て方（安心して使うためのメタルール）

- **スキルは明示指示でしか変わらない** — 会話しているだけでは書き換わらない。「スキルを修正して」「今回の流れに合わせてスキルを調整して」と言ったときだけ変わる。
- **全部 git で追える・戻せる** — いつ誰がどういう意図で変えたかがコミットログに残る。「6/19 時点は良かった」と言えば、おかしくなった原因を遡って特定できる。
- **コンテキストが溜まったらセッションを切る** — 会話が長くなると精度が落ちる。新しいセッションにするとすっきりする。基本のスキルはそのままに、運用が大きく変わったときだけスキルを更新する運用にすると、ゴミが溜まりにくい。
- **判断の記録は private repo（`reality-science-docs`）に溜める** — 「どの議論でこのルールになったか」を残すと、スキル改変の根拠になり追いやすい。

---

## 7. 公開リポと機密

- GitHub Pages 無料運用のため、サイトの repo は **public**。**機密情報を置かない**。
- 議事録・契約・内部資料は **private repo `reality-science-docs`** 側で管理（アクセスは @Chiaki に依頼）。
